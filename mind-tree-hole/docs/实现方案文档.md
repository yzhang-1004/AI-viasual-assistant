## 一、技术选型总览

| 层次 | 技术 | 说明 |
|------|------|------|
| **前端框架** | React 18 + TypeScript + Vite | 组件化开发，类型安全，快速 HMR |
| **样式方案** | Tailwind CSS + CSS Animation | 快速构建 UI，丰富的动画支持 |
| **摄像头/麦克风** | MediaDevices API (`getUserMedia`) | 浏览器原生能力，获取视频流和音频流 |
| **音频处理** | Web Audio API + AudioWorklet | 声波纹可视化、VAD 检测、TTS 播放 |
| **粒子动效** | Canvas 2D API | 萤火虫粒子、呼吸光晕渲染 |
| **WebSocket** | 原生 WebSocket | 连接灵积 ASR/TTS 实时服务 |
| **HTTP 流式** | Fetch API + ReadableStream | 连接灵积 LLM 流式文本生成 |
| **状态管理** | Zustand | 轻量、无模板代码的状态管理 |
| **构建目标** | 纯前端 SPA | 无后端，API Key 由用户配置（本地存储） |

## 三、核心模块实现方案

### 3.1 灵积 LLM 流式调用（`services/llm.ts`）

**接口选型**：使用 **OpenAI 兼容 Chat Completions** 接口，最大程度降低接入成本，支持 `stream: true` 流式返回。

```typescript
// 调用方式伪代码
POST https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions
Headers:
  Authorization: Bearer <DASHSCOPE_API_KEY>
  Content-Type: application/json
Body:
{
  "model": "qwen-plus",          // 或 qwen-max
  "messages": [...],
  "stream": true,                // 启用 SSE 流式
  "temperature": 0.7,
  "max_tokens": 512
}
```

**流式消费**：使用 `fetch` + `ReadableStream` + `TextDecoder` 逐行解析 SSE 事件，每收到一个 `delta.content` 立即通过回调传出。

**关键设计**：
- `stream: true` 实现流式输出，支持 LLM → TTS 的低延迟管道
- 首个 token 到达后，立即将分句文本送入 TTS 合成
- 支持 `stop` 信号中断（AbortController），响应打断机制

---

### 3.2 Fun-ASR 实时语音识别（`services/asr.ts`）

**接口**：WebSocket 双向流式通信。

```
WebSocket URL: wss://dashscope.aliyuncs.com/api-ws/v1/inference
Headers:
  Authorization: Bearer <DASHSCOPE_API_KEY>
```

**交互流程**（5 步）：

```
客户端                               服务端
  |------- WebSocket 握手 ------------>|
  |------- run-task 事件 ------------->|  (携带 model="fun-asr" 等参数)
  |<------ task-started 事件 ----------|
  |                                      |
  |== 循环 ==                            |
  |--- 发送二进制音频数据 (PCM) ------->|
  |<-- result-generated (识别结果) -----|
  |== 循环 ==                            |
  |                                      |
  |------- finish-task 事件 ----------->|
  |<------ task-finished 事件 ----------|
  |------- 关闭 WebSocket ------------->|
```

**关键实现细节**：
- 音频格式：16kHz 采样率，16bit，单声道 PCM
- 端侧使用 `AudioContext` 降采样到 16kHz
- `result-generated` 事件包含实时增量识别文本
- 连接复用：一次性建立连接，持续发送音频，而非每次说话重新连接

**VAD 前置过滤**（`services/vad.ts`）：
- 使用 `AudioWorklet` 在音频线程中实时计算 RMS 能量值
- 能量阈值动态调整（自适应环境噪声）
- 静音段不发送音频数据到 WebSocket，节省带宽和调用成本
- 检测到用户停顿时（>0.8s），触发"预测性预填"逻辑

---

### 3.3 CosyVoice 实时 TTS（`services/tts.ts`）

**接口**：WebSocket 双向流式。

```
WebSocket URL: wss://dashscope.aliyuncs.com/api-ws/v1/inference
Headers:
  Authorization: Bearer <DASHSCOPE_API_KEY>
```

**交互流程**：
```
客户端                               服务端
  |------- run-task (model="cosyvoice") ->|
  |<------ task-started ------------------|
  |--- 发送文本 (text="你好...") -------->|
  |<-- result-generated (PCM 音频) -------|
  |--- 发送文本 (继续...) --------------->|
  |<-- result-generated (PCM 音频) -------|
  |------- finish-task ------------------>|
  |<------ task-finished -----------------|
```

**流式管道设计**：
```
LLM 输出 delta.content
  → 累积到句子边界（遇到 。！？\n 等）
  → 将完整分句发送给 TTS WebSocket
  → TTS 返回 PCM 音频片段
  → AudioContext 解码并立即播放
```

**打断机制**：
- 用户开口 → VAD 检测到语音 → 立即 `AbortController.abort()` 中断 LLM 请求
- 同时关闭 TTS WebSocket 连接（发送 `finish-task`），停止音频播放
- AI 光晕瞬间切换回倾听态

### 3.4 高频话术 TTS 缓存（`services/ttsCache.ts`）

预生成 50 句高频安抚话术的音频文件，存放在 `public/tts-cache/` 目录。

**缓存命中逻辑**：
```typescript
const CACHE_MAP = {
  "深呼吸": "deep-breath.mp3",
  "你很棒": "you-are-great.mp3",
  "我在听": "im-listening.mp3",
  // ... 共 50 句
}

function tryCacheHit(text: string): AudioBuffer | null {
  // 精确匹配 → 直接返回预生成音频
  // 未命中 → 返回 null，走云端 TTS
}
```

**优势**：
- 0 延迟响应（无需等待云端 TTS）
- 节省 TTS API 调用费用
- 高频话术覆盖 80% 的简单共情场景

### 3.5 端侧视觉降维与多模态路由（`services/vision.ts` + `hooks/useVisualAnalysis.ts`）

**策略核心**：不传视频流，只传"关键时刻"的单帧，并将视觉特征文本化。

**处理流程**：

```
摄像头视频流 (30fps)
    │
    ▼ 每 2 秒抽 1 帧（canvas 截图）
    │
    ▼ 端侧轻量分析（TensorFlow.js FaceLandmark / 或简化亮度+运动检测）
    │
    ├─ 无显著变化 → 丢弃，不调用云端
    │
    └─ 有显著变化（表情/亮度/人脸出现消失）
        │
        ├─ 90% 情况：转为文本描述
        │   文本: "[视觉上下文：环境昏暗，用户表情平静]"
        │   → 拼入 LLM System Prompt，调用纯文本 LLM（低成本）
        │
        └─ 10% 情况：用户主动触发多模态
           如 "你看看我现在这样对吗"
           → 将当前帧图片 base64 发送给 Qwen-VL（高成本）
```

**简化版端侧检测**（MVP 阶段不使用 TensorFlow.js，降低复杂度）：
- **亮度检测**：计算 canvas 截图像素的平均亮度值
- **运动检测**：前后两帧像素差异比例
- **人脸检测**：使用浏览器原生 `FaceDetector` API（Chrome 支持）或简化为可选

**多模态触发条件**（进入 L3 路由）：
- 用户明确说"你看"、"你看看"、"这个"、"这样"等指向性词汇
- 用户点击界面"拍照提问"按钮

### 3.6 多级模型路由（`hooks/useConversation.ts` 核心编排）

```
用户语音输入
    │
    ▼
VAD 检测 → 过滤语气词（嗯、啊、哦）
    │
    ▼
ASR 识别文本
    │
    ├─ 文本长度 < 3 字 + 无实质内容
    │   └─ L0 规则引擎 → 直接播放缓存 TTS（"嗯"、"我在听"）
    │      延迟 < 100ms，成本 0
    │
    ├─ 简单寒暄/共情（"今天好累"、"心情不好"）
    │   └─ L2 云端文本 LLM（Qwen-Plus）+ 视觉上下文文本
    │      延迟 1-2s，成本低
    │
    ├─ 复杂心理解惑（"我和老板吵架了怎么办"）
    │   └─ L2 云端文本 LLM（Qwen-Max）+ CBT Prompt + 视觉上下文文本
    │      延迟 1-2s，成本中
    │
    └─ 用户主动要求视觉理解（"你看看我这样对吗"）
        └─ L3 云端多模态（Qwen-VL-Max）+ 当前帧图片
           延迟 2-3s，成本高
```

**路由判断逻辑**：
- L0 触发：文本长度 ≤ 5 字且匹配语气词/简单表达正则
- L2 触发：默认路由，大部分请求
- L3 触发：检测到视觉触发词 OR 用户点击多模态按钮

### 3.7 四种模式的 System Prompt 设计（`utils/prompts.ts`）

**3.7.1 陪伴倾听（默认）**
```
你是一位温暖、善解人意的朋友。你的核心任务是倾听和共情。
规则：
1. 极少给出建议，多用"嗯"、"我在听"、"那一定很难受"等共情回应
2. 回应简短，控制在 1-3 句话
3. 用口语化的、自然的语调
4. 不要分析问题，不要给解决方案
5. 适当使用追问引导用户继续表达
[视觉上下文]：{vision_context}
```

**3.7.2 解惑指路**
```
你是一位温和的认知行为疗法（CBT）咨询师。
规则：
1. 先共情，再引导用户从新的角度看问题
2. 使用 CBT 框架帮助拆解自动负向思维
3. 提供具体可执行的建议，每次最多 2 条
4. 用"我们可以试着这样看..."的而非"你应该..."的语气
5. 回应控制在 3-5 句话
[视觉上下文]：{vision_context}
```

**3.7.3 冥想放松**
```
你是一位专业的正念冥想引导师。
规则：
1. 语速要慢，用词轻柔、舒缓
2. 引导 4-7-8 呼吸法：吸气 4 秒、屏息 7 秒、呼气 8 秒
3. 或引导身体扫描冥想
4. 每次引导一个步骤，等待用户完成后继续
5. 在引导之间加入温和的肯定语
[视觉上下文]：{vision_context}
```

**3.7.4 每日力量**
```
你是一位积极心理学教练。
规则：
1. 基于用户的近期情绪状态，生成专属的每日肯定语
2. 给出一个微小的正向行动建议（"今天试试对镜子里的自己微笑"）
3. 回应温暖而坚定，1-2 句话即可
4. 不追问不展开，就是一句温暖的话
[视觉上下文]：{vision_context}
```

**视觉上下文注入格式**：
```
[视觉上下文：环境{亮度描述}，用户{表情状态描述}]
```
- 纯文本 LLM 调用时拼入 System Prompt 末尾
- 多模态 LLM 调用时以图片形式传入

### 3.8 对话上下文管理（`stores/conversationStore.ts`）

**滑动窗口策略**：
```
[摘要] ← [第1轮] [第2轮] ... [第9轮] ← [第10轮(当前)]
  ↑                                            ↑
  自动摘要压缩                                完整保留
```

**摘要触发条件**：对话轮次 > 10 轮时，将前 9 轮压缩为一段 100 字以内的摘要。

**摘要生成**：调用一次轻量 LLM（Qwen-Turbo），Prompt 为 "将以下对话压缩为一段简短摘要，保留关键情绪和主题"。

**Token 控制**：每轮请求前检查总 token 数，超过阈值则触发摘要压缩，确保每次请求的 context 不超过 2000 tokens。

## 四、UI 组件实现要点

### 4.1 呼吸光晕（`BreathingGlow.tsx`）

**实现方案**：Canvas 2D + requestAnimationFrame

- **核心算法**：多层径向渐变 + 正弦波缩放
  ```
  radius = baseRadius + amplitude * sin(time * frequency)
  ```
- **状态切换**：
  - 倾听态：低频（0.5Hz）、低振幅、暖橙色
  - 说话态：高频（2Hz）、高振幅、涟漪扩散
  - 思考态：中频（1Hz）、轻微颜色闪烁
- **层数**：3-4 层同心渐变，每层不同相位，创造"呼吸感"

### 4.2 萤火虫粒子（`FireflyParticles.tsx`）

**实现方案**：Canvas 2D 粒子系统

- **粒子属性**：x, y, vx, vy, size, opacity, phase
- **运动模型**：布朗运动 + 正弦浮动
  ```
  x += vx + random(-0.5, 0.5)
  y += vy + sin(time + phase) * 0.3
  ```
- **渲染**：径向渐变光点 + 随机闪烁（opacity 正弦变化）
- **粒子数**：30-50 个
- **模式联动**：冥想模式时粒子运动速度降为 1/3

### 4.3 声波纹动画（`MicButton.tsx`）

**实现方案**：Canvas 或 CSS animation

- 使用 `AnalyserNode` 从麦克风流实时获取频率数据
- 绘制 3-5 条水平波动线条，振幅与音量正相关
- 静音时线条平直，说话时波动明显
- 颜色：暖橙色渐变

### 4.4 打字机效果（`AIReplyText.tsx`）

**实现方案**：CSS animation + 逐字显示

- LLM 流式输出每个 token 时，逐字追加到显示文本
- 末尾闪烁光标（CSS `@keyframes blink`）
- 文字从 opacity: 0 淡入（`transition: opacity 0.3s`）
- AI 说完后 3 秒文字逐渐隐去（`opacity → 0`）

### 4.5 冥想呼吸圈（`BreathingCircle.tsx`）

**实现方案**：Canvas 动画

- 一个圆环随时间缩放：
  ```
  scale = 1 + 0.3 * sin(time * (2π / breathCycleSeconds))
  ```
- 4-7-8 呼吸法周期（19秒一个完整循环）：
  - 0-4s：放大（吸气）
  - 4-11s：保持（屏息）
  - 11-19s：缩小（呼气）
- 文字提示："吸气..." / "屏息..." / "呼气..."

```typescript
}
```

### 5.2 核心对话流程（`useConversation` hook）

```
1. 用户说话 → VAD 检测到语音
2. 用户停顿 0.8s → 触发"预测性预填"
3. ASR 文本结果返回
4. 路由判断（L0/L2/L3）
5. L0: 直接返回缓存 TTS
   L2: 发送 LLM 请求（携带视觉上下文文本）
   L3: 截取当前帧 + 发送多模态请求
6. LLM 流式返回 → 逐句送入 TTS
7. TTS 音频片段播放 → 光晕切换为"说话态"
8. 用户打断 → 中断 LLM + 停止 TTS + 光晕回到"倾听态"
9. 对话轮次 +1，检查是否需要摘要压缩
```

## 六、MVP 阶段简化策略

为快速交付可用原型，MVP 阶段做以下简化：

| 完整设计 | MVP 简化 | 说明 |
|---------|---------|------|
| TensorFlow.js 端侧表情检测 | 仅做亮度检测 | 大幅降低前端复杂度 |
| 多级路由 L0-L3 | L0 + L2（省略 L1/L3） | L3 多模态后续迭代再加 |
| 端侧小模型摘要 | 调用 Qwen-Turbo 做摘要 | 摘要频率低，云端成本可控 |
| Fun-ASR WebSocket | 浏览器 Web Speech API（SpeechRecognition） | Chrome 原生支持，零成本快速验证 |
| CosyVoice WebSocket TTS | 浏览器 SpeechSynthesis API + 灵积 TTS HTTP | 先用浏览器 TTS 跑通流程，再切灵积 |
| 50 句缓存 TTS | 5-10 句浏览器 TTS 预生成 | 先用浏览器内置语音验证效果 |
| 萤火虫粒子 | CSS 简化版粒子（伪元素 + animation） | Canvas 粒子系统后续优化 |

**MVP 推荐渐进路线**：
1. **Phase 1**：纯 UI 骨架 + 浏览器 Speech API → 1 天内可交互
2. **Phase 2**：接入灵积 LLM 流式 API → 核心对话闭环
3. **Phase 3**：接入灵积 ASR + TTS WebSocket → 全链路灵积
4. **Phase 4**：Canvas 粒子动效 + 摄像头视觉分析 → 完整体验

## 七、API Key 安全策略

纯前端 SPA 无法安全存储 API Key，提供两种方案：

**方案 A（MVP 推荐）**：用户自行输入
- 设置面板输入灵积 API Key，存储在 `localStorage`
- 每次请求从 localStorage 读取，拼入请求头
- 风险提示：Key 存于浏览器，仅限个人使用

**方案 B（生产推荐）**：简易后端代理
- 部署一个轻量 Cloudflare Worker / Vercel Function 做 API 代理
- 前端不接触 API Key，所有灵积请求走后端转发
- MVP 阶段可跳过

MVP 阶段采用**方案 A**，在设置面板提供 API Key 输入框和获取链接。

## 九、技术风险与注意事项

| 风险 | 应对 |
|------|------|
| 浏览器 Speech API 兼容性差（Firefox/Safari） | 优先验证 Chrome，后续以灵积 ASR 替代 |
| WebSocket 连接可能被防火墙阻止 | 提供 HTTP 降级方案（轮询式 ASR） |
| 灵积 TTS 实时流延迟较大 | 先用浏览器 TTS 兜底，灵积 TTS 用预生成+缓存策略 |
| Canvas 动画性能（低端设备） | 使用 `requestAnimationFrame` + 降低粒子数 |
| `getUserMedia` 需要 HTTPS | 本地开发 `localhost` 豁免，部署时确保 HTTPS |
| API Key 泄露风险 | MVP 期本地存储 + 免责提示，后续上后端代理 |
